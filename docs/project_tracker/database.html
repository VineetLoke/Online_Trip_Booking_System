Database Overview

The Online Booking System uses MongoDB, a NoSQL document database, to store and manage data. MongoDB was chosen for its flexibility, scalability, and ability to handle complex data structures.

Database Schema

The database schema consists of the following collections:

Stores information about registered users of the system.

Schema:
{
  "_id": ObjectId,
  "name": String,
  "email": String,
  "password": String // Hashed,
  "phone": String,
  "address": String,
  "role": String // "user" or "admin",
  "createdAt": Date,
  "updatedAt": Date
}
Indexes:
  • email (unique)
  • phone

Database Design Considerations
MongoDB vs. Relational Database

MongoDB was chosen over a relational database for the following reasons:

  • Flexible Schema: Allows for easy adaptation to changing requirements
  • Document Structure: Natural fit for the hierarchical data in the system
  • Scalability: Horizontal scaling capabilities for future growth
  • Performance: Fast reads and writes for the query patterns in the application
Embedding vs. Referencing

The database design uses a mix of embedding and referencing:

  • Embedding: Used for data that is always accessed together (e.g., traveler information in bookings)
  • Referencing: Used for relationships between major entities (e.g., user to booking)
Indexing Strategy

Indexes are created on fields that are frequently used in queries to improve performance:

  • Single-Field Indexes: For simple queries on one field
  • Compound Indexes: For queries that filter on multiple fields
  • Unique Indexes: To enforce data integrity
Security Considerations

The database design includes several security measures:

  • Password Hashing: User and admin passwords are hashed using bcrypt
  • Access Control: Different roles have different levels of access
  • Data Validation: Schema validation ensures data integrity
